RAG Retrieval-Augmented Generation

정의: LLM이 답변을 생성하기 전에, 외부 지식 저장소에서 관련 정보를 검색(Retrieve)하여 프롬프트에 보강(Augment)한 뒤 생성(Generate)하는 아키텍처 패턴.

왜 RAG가 필요한가?

LLM의 한계: 학습 데이터 이후의 정보를 모르고(시간 제한), 전문 도메인 지식이 부족하며(환각 위험), 대규모 데이터를 파인튜닝하는 것은 비용이 큽니다.

RAG의 해결: 외부 DB에서 최신·정확한 정보를 가져와 LLM에 전달하므로, 파인튜닝 없이도 도메인 특화 답변을 생성할 수 있습니다.

핵심 구성요소

📚 Knowledge Base

원본 데이터(문서, DB, API 등)를 저장하고 검색 가능한 형태로 인덱싱

🧮 Embedding Model

텍스트를 고차원 벡터로 변환하여 의미적 유사도 계산 가능하게 함

🗄️ Vector Store

임베딩 벡터를 저장·검색하는 특화 DB (Chroma, Pinecone, FAISS 등)

🔎 Retriever

질문과 가장 관련 높은 문서를 찾아주는 검색 모듈

📋 Prompt Template

검색 결과 + 사용자 질문을 LLM이 이해할 수 있는 형식으로 조합

🧠 LLM

최종 답변을 생성하는 대규모 언어 모델 (GPT-4, Claude 등)

🎮 프로젝트에서의 RAG 적용 사용중

  • Knowledge Base: Steam 게임 8,301개 데이터 (이름, 장르, 태그, 가격, 리뷰 등)
  • Embedding: text-embedding-3-small (OpenAI)
  • Vector Store: ChromaDB (로컬 영구 저장)
  • Retriever: 하이브리드 검색 — 벡터 유사도 + BM25 키워드 검색을 RRF(Reciprocal Rank Fusion)로 결합
  • LLM: GPT-4o-mini (temperature 0.2)
  • 추가 기법: 쿼리 번역(한→영), 메타데이터 필터 추출, 인기도 기반 재정렬
🔗

LangChain LLM 애플리케이션 프레임워크

정의: LLM을 활용한 애플리케이션을 빠르게 개발할 수 있도록 표준화된 추상화 계층과 컴포넌트 조합 도구를 제공하는 오픈소스 프레임워크. RAG뿐 아니라 챗봇, 에이전트, 요약 등 다양한 LLM 앱을 구축할 수 있다.

RAG와의 관계

RAG는 "무엇을 만들 것인가"(패턴)이고, LangChain은 "어떻게 만들 것인가"(도구)입니다. LangChain 없이도 RAG를 구현할 수 있고, LangChain으로 RAG가 아닌 것도 만들 수 있습니다. 하지만 LangChain은 RAG 구현을 매우 편리하게 해줍니다.

핵심 추상화 (Core Abstractions)

💬 Chat Models

ChatOpenAI, ChatAnthropic 등 — 다양한 LLM을 통일된 인터페이스로 호출

📋 Prompt Templates

ChatPromptTemplate — 변수를 포함한 재사용 가능한 프롬프트 틀

📤 Output Parsers

StrOutputParser, JsonOutputParser — LLM 출력을 원하는 형태로 파싱

🔎 Retrievers

벡터 스토어, BM25 등 다양한 검색 백엔드를 통일된 인터페이스로 제공

🔗 Runnables (LCEL)

RunnableLambda, RunnablePassthrough — 파이프(|) 연산으로 체인을 선언적 구성

📦 Document Loaders

PDF, CSV, 웹페이지 등 다양한 소스에서 데이터를 Document 객체로 로드

✂️ Text Splitters

긴 문서를 적절한 크기의 청크로 분할 (RecursiveCharacterTextSplitter 등)

🧮 Embeddings

OpenAIEmbeddings 등 — 텍스트→벡터 변환 모델의 통합 인터페이스

LCEL (LangChain Expression Language)

LangChain의 핵심 구성 방식. 파이프(|) 연산자로 컴포넌트를 연결하여 선언적으로 체인을 만듭니다:

chain = prompt | llm | output_parser # 기본 체인 rag = retriever | format_docs | prompt | llm # RAG 체인

🎮 프로젝트에서의 LangChain 활용 사용중

  • ChatOpenAI: GPT-4o-mini 모델 호출 (temperature=0.2)
  • ChatPromptTemplate: 시스템 프롬프트, 쿼리 번역, 필터 추출 등 3개 템플릿
  • StrOutputParser: LLM 응답을 문자열로 파싱
  • OpenAIEmbeddings: text-embedding-3-small로 게임 데이터 벡터화
  • Chroma: 벡터 스토어로 사용 (LangChain 통합)
  • RunnableLambda / RunnablePassthrough: LCEL 체인 구성에 활용
🕸️

LangGraph 상태 기반 에이전트 프레임워크

정의: LangChain 팀이 만든 그래프 기반 워크플로우 라이브러리. LLM 애플리케이션에 상태 관리(State), 조건 분기(Conditional Edge), 순환(Cycle)을 추가하여 복잡한 에이전트와 멀티스텝 워크플로우를 구현할 수 있다.

LangChain과의 관계

LangChain의 LCEL은 데이터가 한 방향으로 흐르는 선형 체인(DAG)에 적합합니다. 하지만 "결과를 보고 다시 시도", "여러 도구 중 조건부 선택", "대화 상태 기억" 같은 패턴에는 한계가 있습니다. LangGraph는 이런 순환·분기가 필요한 복잡한 워크플로우를 해결합니다.

핵심 개념 3가지

🔵 Node (노드)

각 처리 단계를 나타내는 함수. LLM 호출, 도구 실행, 데이터 전처리 등 한 가지 작업을 수행

➡️ Edge (엣지)

노드 간 연결. 일반 엣지(항상 이동)와 조건부 엣지(상태에 따라 분기)가 있음

💾 State (상태)

그래프 전체에서 공유되는 딕셔너리. 각 노드가 읽고 수정하며, 대화 기록·중간 결과 등을 저장

그래프 실행 흐름 예시

START
↓
🔍 쿼리 분석
↓조건 분기
📚 RAG 검색
🛠️ 도구 호출
↓
🤖 답변 생성
↓품질 체크 → 불충분 시 재검색
END

🎮 프로젝트에서의 LangGraph 미사용

  • 현재는 단일 턴 추천(질문→검색→답변)이므로 LCEL 체인으로 충분합니다.
  • 도입 시나리오:
  • 🔁 멀티턴 대화 — "방금 추천한 게임 말고 다른 거 알려줘" 같은 맥락 유지
  • 🔀 조건 분기 — 질문 유형(추천/비교/정보)에 따라 다른 처리 경로
  • 🛠️ 도구 사용 — Steam API 실시간 호출, 가격 비교 등 외부 도구 통합
  • ✅ 자체 검증 — 검색 결과가 불충분하면 재검색하는 루프

🎯 언제 무엇을 선택하나?

RAG만 — 직접 구현

간단한 프로토타입이거나 프레임워크 종속을 피하고 싶을 때. OpenAI API + 벡터 DB만으로 충분히 가능.

LangChain + RAG

프로덕션 수준 RAG를 빠르게 구축할 때. 컴포넌트 교체(LLM, DB 등)가 쉽고, 생태계가 풍부.

LangGraph + LangChain + RAG

에이전트, 멀티턴, 도구 호출, 자체 검증 루프 등 복잡한 워크플로우가 필요할 때.

/* 포함 관계 (바깥→안쪽) */ AI / LLM 생태계 └── RAG ← 아키텍처 패턴 (검색 + 생성) ├── LangChain ← 구현 프레임워크 (RAG 외에도 사용 가능) │ └── LangGraph ← 그래프 기반 에이전트 확장 ├── LlamaIndex ← 또 다른 RAG 구현 프레임워크 └── 직접 구현 ← API + 벡터 DB 직접 연결 /* 관계 요약 */ RAG ≠ LangChain ← RAG는 패턴, LangChain은 도구 LangGraph ⊂ LangChain 생태계 ← LangGraph는 LangChain 위에 구축 LangChain ∩ RAG ← LangChain은 RAG 구현에 많이 쓰이지만, 그것만을 위한 건 아님
