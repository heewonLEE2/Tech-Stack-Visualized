RAG 전체 구조

RAG는 외부 지식 검색과 LLM 생성을 결합한 아키텍처 패턴이다. LangChain은 이를 구현하는 프레임워크이고, LangGraph는 에이전트 수준으로 확장한다.

💡 왜 RAG인가?

LLM의 한계: 대규모 언어 모델은 학습 데이터의 시간적 한계(knowledge cutoff)와 도메인 지식 부족이라는 근본적인 문제를 가진다.

해결: RAG는 외부 문서를 실시간으로 검색하여 LLM에 맥락(context)으로 제공한다. 이를 통해 할루시네이션을 줄이고 최신 정보를 활용할 수 있다.

비용 효율: 모델을 재학습(fine-tuning)하는 대신 문서만 업데이트하면 되므로 비용과 시간이 크게 절약된다.

RAG 파이프라인

사용자 질문이 들어오면 Retriever가 관련 문서를 찾고, 이를 프롬프트에 삽입한 뒤 LLM이 답변을 생성한다.

💡 왜 3단계 파이프라인인가?

Retrieve: LLM의 컨텍스트 윈도우는 한정되어 있어 전체 문서를 넣을 수 없다. 관련 문서만 선별하는 과정이 필수적이다.

Augment: 단순히 문서를 넣는 것이 아니라, 질문과 문서를 구조화된 프롬프트로 조합해야 LLM이 효과적으로 활용한다.

Generate: 검색된 맥락을 기반으로 답변을 생성하므로 할루시네이션이 크게 줄어든다.

🐍 LangChain 코드 
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_community.vectorstores import Chroma
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser
from langchain_core.runnables import RunnablePassthrough

# 1. Retrieve
retriever = Chroma.from_documents(docs, OpenAIEmbeddings()).as_retriever(
    search_kwargs={"k": 3}
)

# 2. Augment
prompt = ChatPromptTemplate.from_template(
    "Context: {context}\n\nQuestion: {question}\nAnswer:"
)

# 3. Generate (LCEL chain)
chain = (
    {"context": retriever, "question": RunnablePassthrough()}
    | prompt
    | ChatOpenAI(model="gpt-4")
    | StrOutputParser()
)

result = chain.invoke("RAG란 무엇인가?")

임베딩과 벡터 공간

텍스트를 수치 벡터로 변환하는 임베딩은 RAG의 핵심이다. 의미가 유사한 텍스트는 벡터 공간에서 가까운 위치에 놓인다.

💡 왜 임베딩인가?

의미적 검색: 키워드 매칭은 "강아지"와 "puppy"가 같은 의미임을 모른다. 임베딩은 의미를 수치화하여 유사 개념을 찾을 수 있게 한다.

차원 축소: 자연어의 무한한 표현을 고정 차원(예: 1536차원) 벡터로 압축하여 효율적인 연산이 가능하다.

코사인 유사도: 두 벡터 사이의 각도로 의미적 유사도를 측정한다. 각도가 작을수록(cos θ → 1) 더 유사하다.

🐍 임베딩 코드 
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 단일 텍스트 임베딩
vector = embeddings.embed_query("인공지능이란 무엇인가?")
print(f"차원: {len(vector)}")  # 1536

# 여러 문서 임베딩
doc_vectors = embeddings.embed_documents([
    "머신러닝은 AI의 하위 분야이다.",
    "딥러닝은 신경망 기반 학습이다.",
    "오늘 날씨가 좋다.",
])
# 유사한 문서는 가까운 벡터를 가짐

검색과 랭킹

Retriever는 쿼리와 문서 간 유사도를 계산하여 가장 관련 있는 Top-K 문서를 선택한다.

💡 왜 검색 방식이 중요한가?

Vector Search: 코사인 유사도 기반. 의미적으로 유사한 문서를 잘 찾지만 정확한 키워드 매칭에 약하다.

BM25: 전통적 키워드 기반 검색. 정확한 용어가 포함된 문서를 잘 찾지만 동의어를 모른다.

Hybrid: 두 방식을 결합(Reciprocal Rank Fusion)하여 각각의 약점을 보완한다. 실무에서 가장 높은 성능을 보인다.

검색 방식:
🐍 검색 코드 
from langchain_community.vectorstores import Chroma

vectorstore = Chroma.from_documents(docs, embeddings)

# Vector Search
retriever = vectorstore.as_retriever(
    search_type="similarity",
    search_kwargs={"k": 3}
)

results = retriever.invoke("RAG의 장점은?")
for doc in results:
    print(doc.page_content[:100])

텍스트 청킹

긴 문서를 청크 단위로 분할하여 벡터 스토어에 저장한다. 적절한 크기와 오버랩은 검색 품질에 직접적인 영향을 미친다.

💡 왜 청킹이 필요한가?

컨텍스트 제한: LLM은 한 번에 처리할 수 있는 토큰 수가 제한되어 있다. 문서 전체를 넣을 수 없으므로 적절한 크기로 나눠야 한다.

검색 정밀도: 너무 크면 관련 없는 내용이 섞이고, 너무 작으면 맥락이 손실된다. chunk_size와 chunk_overlap의 균형이 핵심이다.

오버랩: 청크 경계에서 문맥이 잘리는 것을 방지한다. 인접 청크와 일부 텍스트를 공유하여 정보 누락을 줄인다.

🐍 청킹 코드 
from langchain.text_splitter import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=300,
    chunk_overlap=50,
    separators=["\n\n", "\n", ". ", " ", ""],
)

chunks = splitter.split_text(document_text)
print(f"총 {len(chunks)}개 청크 생성")

LangChain과 LCEL

LangChain은 LLM 애플리케이션 개발 프레임워크이다. LCEL(LangChain Expression Language)의 파이프(|) 구문으로 컴포넌트를 연결하여 체인을 구성한다.

💡 왜 LangChain인가?

추상화: LLM, 프롬프트, 검색 등 반복되는 패턴을 표준 인터페이스로 추상화하여 재사용 가능한 컴포넌트로 만든다.

호환성: OpenAI, Anthropic, HuggingFace 등 수십 개 모델과 Chroma, Pinecone, FAISS 등 다양한 벡터 스토어를 동일한 인터페이스로 사용할 수 있다.

LCEL: prompt | llm | parser 형태로 데이터 흐름을 선언적으로 정의한다. 스트리밍, 배치, 비동기를 자동 지원한다.

체인 프리셋:
🐍 LCEL 코드 
# 기본 체인: Prompt → LLM → Parser
from langchain_openai import ChatOpenAI
from langchain_core.prompts import ChatPromptTemplate
from langchain_core.output_parsers import StrOutputParser

prompt = ChatPromptTemplate.from_template(
    "{topic}에 대해 설명해줘."
)
llm = ChatOpenAI(model="gpt-4")
parser = StrOutputParser()

chain = prompt | llm | parser
result = chain.invoke({"topic": "RAG"})

LangGraph 상태 그래프

LangGraph는 노드(함수)와 엣지(흐름)로 구성된 상태 그래프를 정의한다. 조건부 분기와 사이클을 지원하여 복잡한 AI 에이전트를 구현할 수 있다.

💡 왜 LangGraph인가?

LCEL의 한계: LCEL 체인은 선형(A→B→C) 흐름에 최적화되어 있다. 조건 분기(if/else)나 반복(loop)이 필요한 에이전트 워크플로우에는 적합하지 않다.

상태 관리: StateGraph는 실행 중 상태를 공유 딕셔너리로 관리한다. 각 노드가 상태를 읽고 수정하여 다음 노드에 전달한다.

사이클: "도구 호출 → 결과 확인 → 필요 시 재호출" 같은 반복 패턴을 그래프의 사이클로 자연스럽게 표현한다.

그래프 프리셋:
상태(State)가 여기에 표시됩니다.
🐍 LangGraph 코드 
from langgraph.graph import StateGraph, END
from typing import TypedDict

class State(TypedDict):
    question: str
    answer: str

def retrieve(state: State) -> dict:
    docs = retriever.invoke(state["question"])
    return {"context": docs}

def generate(state: State) -> dict:
    answer = llm.invoke(state["context"])
    return {"answer": answer}

# 그래프 구성
graph = StateGraph(State)
graph.add_node("retrieve", retrieve)
graph.add_node("generate", generate)
graph.add_edge("retrieve", "generate")
graph.add_edge("generate", END)
graph.set_entry_point("retrieve")

app = graph.compile()
result = app.invoke({"question": "RAG란?"})

RAG vs LangChain vs LangGraph

세 개념은 서로 다른 레벨에서 동작한다. RAG는 아키텍처 패턴, LangChain은 구현 프레임워크, LangGraph는 에이전트 확장이다.

💡 언제 무엇을 쓸까?

RAG만: 문서 기반 Q&A 같은 단순 검색+생성 태스크. 프레임워크 없이 API 직접 호출로도 구현 가능.

LangChain: 여러 LLM/벡터 스토어를 전환하거나, 프롬프트 관리, 체인 합성이 필요할 때. 빠른 프로토타이핑에 최적.

LangGraph: 멀티턴 대화, 도구 사용 에이전트, 조건부 워크플로우 등 복잡한 흐름 제어가 필요할 때.
