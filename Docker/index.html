Docker & Kubernetes 전체 파이프라인

Dockerfile 작성부터 이미지 빌드, 컨테이너 실행, Docker Compose 오케스트레이션, Kubernetes 배포까지의 전체 흐름입니다. 각 블록을 클릭하면 해당 섹션으로 이동합니다.

💡 왜 컨테이너 기술인가?

환경 일관성: "내 컴퓨터에서는 되는데?"를 해결합니다. 개발·테스트·운영 환경을 동일하게 재현할 수 있습니다.

빠른 배포: VM 대비 컨테이너는 수초 내 시작되어 CI/CD 파이프라인을 극적으로 가속합니다.

효율적 자원 사용: OS 커널을 공유하므로 메모리·디스크 오버헤드가 VM보다 훨씬 적습니다.

컨테이너 vs 가상머신

컨테이너는 하이퍼바이저 기반의 가상머신과 달리 호스트 OS 커널을 공유하여 훨씬 가볍고 빠릅니다. 두 가상화 방식의 구조와 리소스 사용량을 비교해 보세요.

💡 왜 VM 대신 컨테이너인가?

경량화: Guest OS가 없어 메모리 수백 MB 절약, 이미지 크기도 수 MB~수백 MB 수준입니다.

빠른 시작: VM은 부팅에 수십 초 ~ 수 분이 필요하지만, 컨테이너는 밀리초 ~ 수 초 내에 시작됩니다.

밀도: 동일 하드웨어에 수십~수백 개의 컨테이너를 실행할 수 있어 자원 활용도가 높습니다.

비교 모드:
🐳 Docker 명령어 
# 컨테이너 실행 (이미지에서 바로)
docker run -d --name my-app nginx:alpine

# 실행 중인 컨테이너 목록
docker ps

# 컨테이너 리소스 사용량 확인
docker stats

이미지 레이어 구조

Docker 이미지는 레이어들의 스택으로 구성됩니다. Dockerfile의 각 명령어가 새로운 읽기 전용 레이어를 생성하며, 레이어 캐싱을 통해 빌드 속도를 최적화합니다.

💡 왜 레이어 구조인가?

캐싱: 변경되지 않은 레이어는 재사용되어 빌드 시간을 극적으로 단축합니다.

공유: 여러 이미지가 같은 베이스 레이어를 공유하여 디스크 사용량을 절약합니다.

최적화 전략: 자주 변경되는 레이어를 아래쪽에 배치하면 캐시 적중률이 떨어집니다. 변경 빈도가 낮은 순서로 쌓는 것이 핵심입니다.

🐳 Dockerfile 예시 
# Node.js 앱 Dockerfile (최적화)
FROM node:18-alpine          # 베이스 이미지 레이어
WORKDIR /app                 # 메타데이터 (레이어 없음)
COPY package*.json ./        # 의존성 파일만 먼저 복사
RUN npm ci --production      # 의존성 설치 (캐싱 가능)
COPY . .                     # 소스 코드 복사 (자주 변경)
EXPOSE 3000                  # 메타데이터 (레이어 없음)
CMD ["node", "server.js"]    # 메타데이터 (레이어 없음)

컨테이너 라이프사이클

컨테이너는 명확한 생명주기를 가집니다: Created → Running → Paused → Stopped → Removed. 각 상태 전환에 대응하는 Docker CLI 명령어와 함께 전체 흐름을 확인하세요.

💡 왜 라이프사이클을 이해해야 하나?

리소스 관리: 정지된 컨테이너도 디스크를 차지합니다. 불필요한 컨테이너를 제거하지 않으면 디스크 낭비가 누적됩니다.

데이터 영속성: 컨테이너 삭제 시 내부 데이터가 사라집니다. 중요한 데이터는 볼륨으로 분리해야 합니다.

디버깅: 컨테이너 상태를 정확히 파악해야 문제 원인을 빠르게 진단할 수 있습니다.

🐳 Docker 명령어 
# 컨테이너 생성 (실행 안 함)
docker create --name my-app nginx

# 컨테이너 시작
docker start my-app

# 컨테이너 일시정지 / 재개
docker pause my-app
docker unpause my-app

# 컨테이너 정지 (SIGTERM → 10s → SIGKILL)
docker stop my-app

# 컨테이너 강제 종료 (SIGKILL)
docker kill my-app

# 컨테이너 삭제
docker rm my-app

# 실행 + 삭제 (1회성 작업)
docker run --rm nginx echo "hello"

볼륨 & 네트워크

볼륨은 컨테이너와 독립적으로 데이터를 영속화하고, Docker 네트워크는 컨테이너 간 통신을 가능하게 합니다. Bridge, Host, Overlay 네트워크 모드를 비교해 보세요.

💡 왜 볼륨과 네트워크를 분리하나?

데이터 영속성: 컨테이너 삭제 시 내부 파일은 사라지지만, 볼륨의 데이터는 독립적으로 보존됩니다.

서비스 격리: 네트워크를 분리하면 서비스 간 불필요한 통신을 차단하여 보안을 강화합니다.

마이크로서비스: 각 서비스가 독립적인 컨테이너에서 실행되므로, 네트워크 설정으로 서비스 디스커버리와 라우팅을 관리합니다.

🐳 Docker 명령어 
# === 볼륨 ===
# Named Volume 생성
docker volume create my-data

# 볼륨 마운트하여 실행
docker run -v my-data:/app/data nginx

# Bind Mount (호스트 경로 직접 연결)
docker run -v $(pwd)/src:/app/src nginx

# tmpfs (메모리 임시 저장)
docker run --tmpfs /tmp nginx

# === 네트워크 ===
# 네트워크 생성
docker network create my-net

# 네트워크에 연결하여 실행
docker run --network my-net --name web nginx

# 포트 매핑 (-p 호스트:컨테이너)
docker run -p 8080:80 nginx

Docker Compose

Docker Compose는 다중 컨테이너 애플리케이션을 하나의 YAML 파일로 정의하고 관리합니다. 서비스 간 의존성, 네트워크, 볼륨을 선언적으로 구성하여 한 번에 배포할 수 있습니다.

💡 왜 Docker Compose인가?

복잡성 해소: web + API + DB + cache를 각각 docker run하는 대신, 하나의 YAML로 전체 스택을 정의합니다.

재현 가능성: docker-compose.yml 파일만 있으면 누구나 동일한 환경을 즉시 구성할 수 있습니다.

개발 생산성: docker compose up 한 명령으로 전체 개발 환경이 수 초 만에 준비됩니다.

🐳 docker-compose.yml 
version: '3.8'
services:
  web:
    image: nginx:alpine
    ports:
      - "80:80"
    depends_on:
      - app
    networks:
      - frontend

  app:
    build: ./app
    environment:
      - DB_HOST=db
      - REDIS_HOST=cache
    depends_on:
      - db
      - cache
    networks:
      - frontend
      - backend

  db:
    image: postgres:15-alpine
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      - POSTGRES_PASSWORD=secret
    networks:
      - backend

  cache:
    image: redis:7-alpine
    networks:
      - backend

volumes:
  db-data:

networks:
  frontend:
  backend:

Kubernetes 아키텍처

Kubernetes 클러스터는 Control Plane(마스터)과 Worker Node로 구성됩니다. API Server를 중심으로 모든 요청이 흐르며, etcd에 상태가 저장됩니다.

💡 왜 Kubernetes인가?

자동 복구: 컨테이너가 죽으면 자동으로 재시작하고, 노드 장애 시 다른 노드에 자동 재스케줄링합니다.

선언적 관리: "3개의 복제본이 실행되어야 한다"고 선언하면 K8s가 현재 상태를 원하는 상태로 맞춰줍니다 (Desired State).

수평 확장: 트래픽 증가 시 자동으로 Pod를 추가하고, 감소 시 축소합니다 (HPA).

⎈ kubectl 명령어 
# 클러스터 정보 확인
kubectl cluster-info

# 노드 목록
kubectl get nodes

# Pod 목록 (전체 네임스페이스)
kubectl get pods -A

# Deployment 생성
kubectl create deployment nginx --image=nginx:alpine

# 서비스 노출
kubectl expose deployment nginx --port=80 --type=LoadBalancer

# YAML로 리소스 적용
kubectl apply -f deployment.yaml

# Pod 로그 확인
kubectl logs pod-name

# Pod 내부 접속
kubectl exec -it pod-name -- /bin/sh

배포 & 스케일링

Deployment는 ReplicaSet을 관리하며 Rolling Update로 무중단 배포를 수행합니다. HPA를 사용해 트래픽에 따라 자동으로 Pod를 스케일링합니다.

💡 왜 Rolling Update인가?

무중단 배포: 기존 Pod를 하나씩 새 버전으로 교체하여 서비스 중단 없이 배포합니다.

롤백: 새 버전에 문제가 발생하면 이전 버전으로 즉시 롤백할 수 있습니다.

HPA (Horizontal Pod Autoscaler): CPU/메모리 사용률을 모니터링하여 자동으로 Pod 수를 조절합니다.

레플리카 수: 3
⎈ kubectl 명령어 
# 스케일링
kubectl scale deployment my-app --replicas=5

# Rolling Update (이미지 변경)
kubectl set image deployment/my-app app=my-app:v2

# 업데이트 상태 확인
kubectl rollout status deployment/my-app

# 롤백 (이전 버전으로)
kubectl rollout undo deployment/my-app

# HPA 설정 (CPU 50% 기준, 2~10개)
kubectl autoscale deployment my-app \
  --cpu-percent=50 --min=2 --max=10

# HPA 상태 확인
kubectl get hpa
