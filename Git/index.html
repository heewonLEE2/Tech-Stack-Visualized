Git 전체 워크플로우

Git의 핵심 영역인 Working Directory → Staging Area → Local Repository → Remote Repository 전체 데이터 흐름입니다. 각 블록을 클릭하면 해당 섹션으로 이동합니다.

💡 왜 버전 관리인가?

이전으로 되돌리기: 코드의 모든 변경 이력을 저장하여, 문제가 발생하면 언제든 이전 상태로 안전하게 복원할 수 있습니다.

동시 협업: 여러 개발자가 같은 코드베이스에서 독립적으로 작업한 뒤 충돌 없이 병합할 수 있습니다.

분산형 아키텍처: Git은 중앙 서버 없이도 각 개발자가 전체 히스토리를 가지므로, 오프라인에서도 커밋·브랜치·머지가 가능합니다.

Git 내부 객체 구조

Git은 내부적으로 4가지 핵심 객체를 사용합니다: Blob(파일 내용), Tree(디렉토리 구조), Commit(스냅샷), Tag(태그). 모든 객체는 SHA-1 해시로 식별됩니다.

💡 왜 해시 기반 저장인가?

무결성 보장: 내용이 1비트라도 바뀌면 해시가 완전히 달라지므로, 데이터 손상을 즉시 감지합니다.

효율적 중복 제거: 동일 내용의 파일은 같은 해시를 가지므로, 하나의 blob만 저장하여 저장 공간을 절약합니다.

불변 데이터: 한 번 생성된 객체는 변경되지 않아 히스토리의 신뢰성이 보장됩니다.

SHA-1 해시 데모: —
탐색할 객체:
💻 Git 명령어 
# 객체 내용 확인
git cat-file -p <sha>

# 객체 타입 확인
git cat-file -t <sha>

# 파일로부터 해시 생성
echo "hello" | git hash-object --stdin

브랜치 (Branch)

브랜치는 특정 커밋을 가리키는 가벼운 포인터입니다. HEAD는 현재 작업 중인 브랜치를 가리킵니다. 브랜치를 만들고 전환하며 커밋 그래프가 어떻게 변화하는지 확인해 보세요.

💡 왜 브랜치인가?

독립적 작업 공간: 새 기능 개발, 버그 수정을 메인 코드에 영향 주지 않고 독립적으로 진행합니다.

가벼운 구현: Git 브랜치는 40자리 해시 하나만 저장하는 파일이라 생성 비용이 거의 없어 자유롭게 만들고 삭제할 수 있습니다.

병렬 실험: 여러 접근법을 동시에 시도한 후 최적의 결과만 병합할 수 있습니다.

💻 Git 명령어 
# 브랜치 목록 확인
git branch

# 새 브랜치 생성
git branch feature/login

# 브랜치 전환
git switch feature/login   # 또는 git checkout feature/login

# 생성과 동시에 전환
git switch -c feature/login

# 브랜치 삭제
git branch -d feature/login

Merge & Rebase

두 브랜치를 합치는 방법: Merge는 히스토리를 보존하고, Rebase는 히스토리를 깔끔하게 정리합니다. Fast-forward와 3-way merge의 차이를 비교해 보세요.

💡 Merge vs Rebase, 언제 쓰나?

Merge: 공유 브랜치(main)에서는 히스토리 보존이 중요하므로 merge를 사용합니다. 협업 시 안전합니다.

Rebase: 개인 브랜치에서는 rebase로 히스토리를 깔끔하게 만든 뒤 merge합니다. 단, 이미 push한 커밋은 rebase하면 안 됩니다.

Fast-forward: 분기가 없으면 merge commit 없이 포인터만 이동합니다. --no-ff로 강제 merge commit을 만들 수 있습니다.

모드:
💻 Git 명령어 
# Fast-forward merge
git merge feature/login

# Merge commit 강제 생성
git merge --no-ff feature/login

# Rebase (현재 브랜치를 main 위로 재배치)
git rebase main

# 충돌 해결 후 계속
git rebase --continue

# Rebase 취소
git rebase --abort

원격 저장소 (Remote)

원격 저장소는 네트워크를 통해 접근하는 공유 저장소입니다. clone, fetch, pull, push 명령으로 로컬과 원격을 동기화합니다.

💡 왜 원격 저장소인가?

백업: 로컬 코드의 안전한 사본을 원격에 보관합니다.

협업 허브: 팀원들이 같은 원격 저장소를 공유하여 코드 리뷰, PR, 이슈 관리를 체계적으로 수행합니다.

fetch vs pull: fetch는 원격 변경을 다운로드만 하고, pull은 다운로드 + 현재 브랜치에 병합까지 수행합니다.

💻 Git 명령어 
# 원격 저장소 복제
git clone https://github.com/user/repo.git

# 원격 추가
git remote add origin https://github.com/user/repo.git

# 원격 변경사항 다운로드 (병합 X)
git fetch origin

# 원격 변경사항 다운로드 + 병합
git pull origin main

# 로컬 변경사항 업로드
git push origin main

# 원격 브랜치 목록
git branch -r

GitHub 워크플로우

GitHub의 표준 협업 사이클: Fork → Branch → Commit → Pull Request → Code Review → Merge. 오픈소스 기여와 팀 협업의 핵심 패턴입니다.

💡 왜 GitHub Flow인가?

코드 품질: PR을 통한 코드 리뷰로 버그를 조기에 발견하고 코드 품질을 유지합니다.

문서화된 변경: 모든 변경에 설명, 토론, 승인 기록이 남아 왜 이 코드가 추가되었는지 추적 가능합니다.

CI/CD 연동: PR 생성 시 자동 테스트·빌드가 실행되어 안전한 배포를 보장합니다.

💻 GitHub CLI 명령어 
# Fork (GitHub 웹 또는 CLI)
gh repo fork owner/repo

# 브랜치 생성 후 작업
git switch -c feature/new-feature

# 변경 사항 커밋
git add .
git commit -m "feat: add new feature"

# Push 후 PR 생성
git push origin feature/new-feature
gh pr create --title "Add new feature" --body "설명..."

# PR 머지 (리뷰 승인 후)
gh pr merge 42

고급 명령어

reset의 3가지 모드(soft, mixed, hard), revert, cherry-pick, stash, reflog의 동작 원리를 시각적으로 비교합니다.

💡 reset vs revert, 언제 쓰나?

reset: 히스토리를 되감아서 커밋을 제거합니다. 개인 브랜치에서만 사용해야 합니다 (push 전).

revert: 이전 커밋을 취소하는 새 커밋을 만듭니다. 공유 브랜치에서 안전하게 변경을 되돌릴 때 사용합니다.

cherry-pick: 다른 브랜치의 특정 커밋 하나만 골라서 적용할 때 사용합니다.

명령어:
reset 모드:
💻 Git 명령어 
# reset (3가지 모드)
git reset --soft HEAD~1   # 커밋만 취소, 변경사항 staged 유지
git reset --mixed HEAD~1  # 커밋 취소 + unstage, 파일은 유지
git reset --hard HEAD~1   # 커밋 취소 + 변경사항 완전 삭제

# revert (안전한 취소)
git revert HEAD           # 마지막 커밋을 취소하는 새 커밋 생성

# cherry-pick (특정 커밋 가져오기)
git cherry-pick abc1234

# stash (임시 저장)
git stash                 # 변경사항 임시 저장
git stash pop             # 가장 최근 stash 복원
git stash list            # stash 목록 확인

# reflog (복구용 히스토리)
git reflog                # HEAD 이동 히스토리 확인

Git 실전 시뮬레이터

Git 명령어를 직접 입력하면 커밋 그래프가 실시간으로 변화합니다. 아래 명령어를 사용해 브랜치 생성, 커밋, 병합을 연습해 보세요.

💡 사용 가능한 명령어

git commit -m "메시지" — 새 커밋 생성

git branch 이름 — 새 브랜치 생성

git switch 이름 — 브랜치 전환

git merge 이름 — 브랜치 병합

git rebase 이름 — 리베이스

git reset --soft/--mixed/--hard HEAD~N — 리셋

git revert HEAD — 마지막 커밋 되돌리기

git stash / git stash pop — 임시 저장
