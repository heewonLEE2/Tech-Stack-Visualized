AWS Cloud 전체 아키텍처

클라이언트 요청이 Route 53(DNS)을 거쳐 CloudFront(CDN)로 전달되고, ALB가 트래픽을 분산하여 EC2 또는 Lambda에서 처리한 뒤, RDS / DynamoDB에서 데이터를 조회하고 S3에 저장하는 전체 흐름입니다. 각 블록을 클릭하면 해당 섹션으로 이동합니다.

💡 왜 클라우드를 쓰는가?

탄력성(Elasticity): 트래픽 급증 시 자동으로 리소스를 확장하고, 감소 시 축소하여 비용을 최적화합니다.

글로벌 확장: 전 세계 30개 이상의 리전에서 수 분 만에 서비스를 배포할 수 있습니다.

운영 부담 감소: 하드웨어 관리, 패치, 백업 등을 AWS가 대신 처리하여 비즈니스 로직에 집중할 수 있습니다.

글로벌 인프라스트럭처

AWS는 전 세계 리전(Region)에 분산 배치된 가용 영역(AZ)으로 구성됩니다. 각 리전은 물리적으로 격리된 2~6개의 AZ를 포함하며, 엣지 로케이션은 CloudFront CDN을 통해 콘텐츠를 사용자 가까이 전달합니다.

💡 왜 리전과 AZ를 이해해야 하나?

지연시간: 사용자와 가까운 리전을 선택하면 네트워크 지연을 최소화합니다.

고가용성: 다중 AZ에 배포하면 하나의 데이터센터 장애에도 서비스가 중단되지 않습니다.

규정 준수: 데이터 주권 법률에 따라 특정 리전에 데이터를 보관해야 할 수 있습니다.

배포 모드:
☁ AWS CLI 
# 사용 가능한 리전 목록
aws ec2 describe-regions --output table

# 특정 리전의 가용 영역 확인
aws ec2 describe-availability-zones \
  --region ap-northeast-2

# CloudFront 엣지 로케이션 확인
aws cloudfront list-distributions

IAM (Identity & Access Management)

IAM은 AWS 리소스에 대한 인증과 인가를 관리합니다. 사용자, 그룹, 역할, 정책의 계층 구조를 통해 최소 권한 원칙을 적용합니다.

💡 왜 IAM이 중요한가?

보안 기본: 모든 AWS 접근은 IAM을 통해 인증되며, 잘못된 IAM 설정이 보안 사고의 주요 원인입니다.

최소 권한 원칙: 필요한 권한만 부여하면 피해 범위를 최소화할 수 있습니다.

감사 추적: CloudTrail과 연동하여 누가 언제 무엇을 했는지 추적할 수 있습니다.

☁ AWS CLI 
# IAM 사용자 생성
aws iam create-user --user-name developer

# 그룹에 사용자 추가
aws iam add-user-to-group \
  --user-name developer --group-name dev-team

# 정책 연결 (S3 읽기 전용)
aws iam attach-user-policy \
  --user-name developer \
  --policy-arn arn:aws:iam::aws:policy/AmazonS3ReadOnlyAccess

# 역할 생성 (EC2용)
aws iam create-role \
  --role-name EC2-S3-Access \
  --assume-role-policy-document file://trust-policy.json

# 현재 자격 증명 확인
aws sts get-caller-identity

VPC & 네트워킹

VPC는 AWS 클라우드 내에서 논리적으로 격리된 가상 네트워크입니다. 서브넷, 인터넷 게이트웨이, NAT 게이트웨이, 보안 그룹, NACL을 조합하여 세밀한 네트워크 아키텍처를 설계합니다.

💡 왜 VPC가 필요한가?

네트워크 격리: 리소스를 외부로부터 논리적으로 격리하여 보안을 강화합니다.

세밀한 제어: 서브넷, 라우팅 테이블, 보안 그룹으로 트래픽 흐름을 정밀하게 제어합니다.

다중 계층 아키텍처: Public/Private 서브넷을 분리하여 웹 서버와 DB 서버의 보안 수준을 차등 적용합니다.

☁ AWS CLI 
# VPC 생성 (10.0.0.0/16)
aws ec2 create-vpc --cidr-block 10.0.0.0/16

# 퍼블릭 서브넷 생성
aws ec2 create-subnet \
  --vpc-id vpc-xxx --cidr-block 10.0.1.0/24 \
  --availability-zone ap-northeast-2a

# 프라이빗 서브넷 생성
aws ec2 create-subnet \
  --vpc-id vpc-xxx --cidr-block 10.0.2.0/24 \
  --availability-zone ap-northeast-2a

# 인터넷 게이트웨이 생성 & 연결
aws ec2 create-internet-gateway
aws ec2 attach-internet-gateway \
  --internet-gateway-id igw-xxx --vpc-id vpc-xxx

# 보안 그룹 생성
aws ec2 create-security-group \
  --group-name web-sg --description "Web Server SG" \
  --vpc-id vpc-xxx

# 인바운드 규칙 추가 (HTTP 허용)
aws ec2 authorize-security-group-ingress \
  --group-id sg-xxx --protocol tcp \
  --port 80 --cidr 0.0.0.0/0

EC2 & 컴퓨팅

EC2는 AWS의 핵심 컴퓨팅 서비스입니다. 인스턴스 타입을 선택하고, Auto Scaling Group으로 자동 확장/축소를 설정하며, Elastic Load Balancer로 트래픽을 분산합니다.

💡 왜 Auto Scaling인가?

가용성: 인스턴스 장애 시 자동으로 새 인스턴스를 시작하여 서비스 연속성을 보장합니다.

비용 최적화: 트래픽이 적을 때는 인스턴스를 줄여 불필요한 비용을 절감합니다.

무중단 배포: ELB와 결합하면 새 인스턴스로 점진적으로 트래픽을 전환하여 다운타임 없이 배포할 수 있습니다.

트래픽 수준: 3
☁ AWS CLI 
# EC2 인스턴스 시작
aws ec2 run-instances \
  --image-id ami-0c55b159cbfafe1f0 \
  --instance-type t3.micro \
  --key-name my-key --count 1

# Auto Scaling Group 생성
aws autoscaling create-auto-scaling-group \
  --auto-scaling-group-name my-asg \
  --launch-template LaunchTemplateName=my-lt \
  --min-size 2 --max-size 10 --desired-capacity 3 \
  --vpc-zone-identifier "subnet-xxx,subnet-yyy"

# 대상 추적 스케일링 정책
aws autoscaling put-scaling-policy \
  --auto-scaling-group-name my-asg \
  --policy-name cpu-target-tracking \
  --policy-type TargetTrackingScaling \
  --target-tracking-configuration \
    "PredefinedMetricSpecification={PredefinedMetricType=ASGAverageCPUUtilization},TargetValue=50.0"

# ALB 생성
aws elbv2 create-load-balancer \
  --name my-alb --type application \
  --subnets subnet-xxx subnet-yyy

S3 & 스토리지

Amazon S3는 무제한 확장이 가능한 객체 스토리지 서비스입니다. 스토리지 클래스를 선택하여 비용을 최적화하고, 라이프사이클 정책으로 데이터의 자동 이동과 만료를 관리합니다.

💡 왜 스토리지 클래스를 구분하나?

비용 최적화: 자주 접근하는 데이터와 아카이브 데이터의 저장 비용이 최대 95% 차이납니다.

자동화: 라이프사이클 정책으로 시간 경과에 따라 자동으로 저렴한 클래스로 이동합니다.

내구성: 모든 S3 클래스는 99.999999999%(11 Nine) 내구성을 보장합니다.

☁ AWS CLI 
# S3 버킷 생성
aws s3 mb s3://my-bucket-name \
  --region ap-northeast-2

# 파일 업로드
aws s3 cp myfile.txt s3://my-bucket-name/

# 스토리지 클래스 지정하여 업로드
aws s3 cp archive.zip s3://my-bucket-name/ \
  --storage-class GLACIER

# 라이프사이클 정책 설정
aws s3api put-bucket-lifecycle-configuration \
  --bucket my-bucket-name \
  --lifecycle-configuration '{
    "Rules": [{
      "ID": "MoveToIA",
      "Status": "Enabled",
      "Transitions": [{
        "Days": 30,
        "StorageClass": "STANDARD_IA"
      }, {
        "Days": 90,
        "StorageClass": "GLACIER"
      }]
    }]
  }'

# 버킷 내용 조회
aws s3 ls s3://my-bucket-name/ --recursive

데이터베이스 서비스

AWS는 용도에 맞는 다양한 관리형 데이터베이스를 제공합니다. RDS(관계형), DynamoDB(NoSQL), ElastiCache(인메모리 캐시)의 아키텍처와 사용 사례를 비교합니다.

💡 왜 관리형 DB인가?

운영 부담 감소: 패치, 백업, 복제 등을 AWS가 자동으로 처리합니다.

고가용성: Multi-AZ 배포로 자동 장애 조치(Failover)를 지원합니다.

읽기 확장: Read Replica를 추가하여 읽기 트래픽을 수평 확장할 수 있습니다.

☁ AWS CLI 
# RDS MySQL 인스턴스 생성 (Multi-AZ)
aws rds create-db-instance \
  --db-instance-identifier my-db \
  --db-instance-class db.t3.micro \
  --engine mysql --master-username admin \
  --master-user-password secret123 \
  --allocated-storage 20 --multi-az

# Read Replica 생성
aws rds create-db-instance-read-replica \
  --db-instance-identifier my-db-replica \
  --source-db-instance-identifier my-db

# DynamoDB 테이블 생성
aws dynamodb create-table \
  --table-name Users \
  --attribute-definitions \
    AttributeName=userId,AttributeType=S \
  --key-schema \
    AttributeName=userId,KeyType=HASH \
  --billing-mode PAY_PER_REQUEST

# ElastiCache Redis 클러스터 생성
aws elasticache create-cache-cluster \
  --cache-cluster-id my-cache \
  --engine redis \
  --cache-node-type cache.t3.micro \
  --num-cache-nodes 1

서버리스 아키텍처

Lambda는 서버 관리 없이 코드를 실행하는 서버리스 컴퓨팅 서비스입니다. API Gateway로 HTTP 엔드포인트를 생성하고, Step Functions로 워크플로우를 오케스트레이션합니다. 이벤트 드리븐 아키텍처의 핵심입니다.

💡 왜 서버리스인가?

운영 부담 제거: OS 패치, 스케일링, 가용성을 AWS가 100% 관리합니다.

이벤트 단위 과금: 요청이 없으면 비용이 0원이며, 실행 시간(ms) 단위로만 과금됩니다.

자동 확장: 동시 요청 수에 따라 수천 개의 함수가 병렬로 실행됩니다.

☁ AWS CLI 
# Lambda 함수 생성
aws lambda create-function \
  --function-name my-api \
  --runtime python3.12 \
  --handler lambda_function.handler \
  --role arn:aws:iam::123456789012:role/lambda-role \
  --zip-file fileb://function.zip

# API Gateway REST API 생성
aws apigateway create-rest-api \
  --name "My API" \
  --description "Serverless API"

# Step Functions 상태 머신 생성
aws stepfunctions create-state-machine \
  --name "OrderWorkflow" \
  --definition file://workflow.json \
  --role-arn arn:aws:iam::123456789012:role/step-role

# Lambda 함수 호출 테스트
aws lambda invoke \
  --function-name my-api \
  --payload '{"key": "value"}' \
  output.json

# CloudWatch 로그 확인
aws logs filter-log-events \
  --log-group-name /aws/lambda/my-api
